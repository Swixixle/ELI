## 0) One critical idempotency detail to verify

There are **two** ingest patterns:

1. **Append-to-existing case** (Lantern already knows `caseId`)
2. **Create-or-append** (Lantern does *not* know a caseId yet, so ingest creates a new case)

If your dedupe check only searches **events inside a given case**, then:

* ✅ it will stop double-click duplicates **when `caseId` is provided**
* ❌ it will *not* stop duplicates **when no `caseId` is provided**, because each retry would create a brand-new case before you even have a case to search

Your smoke test description (“case created… retry rejected”) *suggests* you already dedupe **before case creation** (global), which is the correct implementation — but it’s worth confirming explicitly.

### The definitive test (run this)

Call ingest **twice** with the **same `source + requestId`** and **NO `caseId`**:

* 1st call: should create case + return `{ caseId, url }`
* 2nd call: should return **409 DUPLICATE_REQUEST** (and NOT create a new case)

If that’s true, your idempotency is real.

If it’s not true, the correct fix is a small “dedupe ledger” with a **unique constraint** on `(source, requestId)` (more on that below).

## 3) If your current dedupe is not global, here’s the correct “bulletproof” version

If the test above creates two cases instead of returning 409, do this:

### Add a tiny dedupe table

`ingest_requests`:

* `source` (text)
* `request_id` (text)
* `case_id` (text)
* `created_at`
* **UNIQUE(source, request_id)**

Flow:

* On ingest: try insert `(source, requestId)` first

  * if unique violation → return 409 with existing `case_id`
  * else continue, create case/event, then update row with `case_id`

That’s the cleanest enterprise-proof way and it’s a small diff.
