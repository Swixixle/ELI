A) UI: Rename and wire the action as “Seal”

	•	Primary button label: Seal
	•	Confirmation modal title: Seal Artifact

	•	Buttons: Seal (primary), Cancel (secondary)

B) API: Seal an artifact (atomic operation)

Create or confirm a single endpoint (preferred):
	•	POST /api/cases/:id/seal

This endpoint must atomically:
	•	freeze the current case state
	•	compute required hashes
	•	apply cryptographic signature (if configured)
	•	persist the sealed artifact as immutable
	•	return identifiers required for retrieval

Response (machine-only):
	•	artifactId
	•	caseId
	•	issuedAt
	•	contentHash (SHA-256)
	•	caseStateHash (SHA-256)
	•	signatureB64 (if signed)
	•	publicKeyId (if signed)
	•	sealStatus: SEALED

Errors:
	•	Return 409 if sealing is blocked by case state
	•	No explanatory text; use stable error keys only

⸻

C) Retrieval: Sealed artifacts must be retrievable

Implement or confirm the following endpoints with no interpretation:
	1.	GET /api/printouts/:id
Returns the sealed artifact payload and immutable metadata only:

	•	identifiers

	•	hashes
	•	signature fields
	•	content or stable content reference

	2.	GET /api/cases/:id/printouts
Returns a list of sealed artifacts for the case:

	•	artifactId
	•	issuedAt
	•	verificationStatus only

No summaries, scores, or judgments.
	3.	Keep or confirm verification endpoint:

Returns cryptographic verification fields only.
