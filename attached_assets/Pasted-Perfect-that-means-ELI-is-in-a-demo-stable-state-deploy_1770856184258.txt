### Phase 2 — “Send to ELI” button (light integration, real value)

Goal: After Lantern produces an output, you can click **Send to ELI** and create an ELI case event (or case stub) automatically.

**Minimal design (recommended):**

* Lantern calls **a new ELI endpoint** that creates an event (or case + event).
* ELI returns `caseId` + link to the case.

**ELI endpoint example**
`POST /api/lantern/ingest`

Body:

```json
{
  "source": "lantern",
  "requestId": "uuid",
  "eventTime": "2026-02-11T18:22:00.000Z",
  "text": "... lantern output ...",
  "metadata": { "model": "gpt-...", "threadId": "...", "userTag": "..." }
}
```

Auth (simple and sufficient for now):

* `Authorization: Bearer <INGEST_TOKEN>`
* Store token in Lantern secrets + ELI secrets.

**What ELI does:**

* Validate schema
* Create an event using the same `{ eventType, eventTime, description, metadata }` pattern you already hardened
* Append an audit event (so the ingest is forensically logged)
* Return `{ caseId, url }`

This is the first point where “integration” starts paying off.

### Phase 3 — Evidence-grade integration (your real differentiator)

Once Phase 2 is stable:

* Lantern generates an **evidence capsule** (transcript hash + payload + signature or receipt)
* ELI stores *only the receipt artifacts* or references
* ELI can verify integrity without needing Lantern’s DB

This is where your HALO/receipts work becomes the backbone, not just a feature.
