The High-Impact Fixes (You Should Make)

These are specific and worth changing because they could create real downstream confusion or risk.

1) “DELETE /api/cases/:id” contradicts “No (soft delete)” in Mutability Rules

You say:
	•	Case delete: “No (soft delete)”
But Endpoint Map says:
	•	DELETE /api/cases/:id → “DELETE case + cascades”

This is a real contradiction.

Fix (recommended):
	•	Change endpoint semantics to PATCH /api/cases/:id { status: 'closed' | 'archived' }
	•	Or keep DELETE but document it as soft delete (status flag) and explicitly state what “cascade” means (if anything is actually physically deleted).

Auditors will notice this mismatch immediately.

⸻

2) “Canon Document (Evidence)” is conceptually dangerous wording

You’re using “Canon document” to mean authoritative rule sets uploaded to a case. That’s fine.

But labeling it “Evidence” risks conflating:
	•	Governing standard (policy/rule)
with
	•	Evidence of facts (records, logs, testimony, third-party verification)

In regulated settings, that distinction matters.

Fix (recommended):
	•	Rename conceptually in the doc:
	•	“Canon Document (Governing Standard)” or “Canon Source (Authority)”
	•	If you also store “case evidence documents,” separate them as a different entity or at least a different type classification.

Right now, someone could assume ELI treats Canon as “evidence,” which is not what you mean.

⸻

3) Public dataset integration conflicts with “Citation Requirements” framing

You state citations must come from:
	•	Canon documents OR
	•	Public datasets

But later you list “Public dataset integration” as “Planned (Not Started).”

If datasets are planned, your “citation requirements” should reflect current reality:
	•	“Citations must come from Canon documents. Dataset citations are supported in the model but not yet integrated.”

Otherwise reviewers will ask, “Where are these datasets now?”

⸻

4) Medical safety flag seems out of scope for ELI (unless justified)

You list medical_safety as an “Immediate refusal.”

But ELI’s domain is procedural admissibility for governance decisions, not medical advice. If this is present because the system can be asked anything, that’s fine—however it needs a framing sentence:

Fix (recommended):
	•	Add one line:
“Because ELI may receive user input outside governance scope, certain high-risk domains (e.g., medical advice) trigger immediate refusal regardless of case context.”

Otherwise it reads like ELI is doing medical adjudication, which contradicts your positioning.

⸻

5) Determination status types are inconsistent in two places

You have:
	•	Determination: 'permitted' | 'blocked' | 'advisory'
But earlier in Canon threshold language you use:
	•	“Review permitted — high procedural risk”
	•	“Review strong — defensible”
	•	“Review robust — regulator-ready”

Those are different vocabularies.

Fix (recommended):
Define a single canonical set of fields, e.g.:
	•	reviewPermission: advisory_only | permitted
	•	proceduralRiskTier: 0_2 | 3 | 4 | 5 or unsafe | high_risk | defensible | regulator_ready

This avoids ambiguity where “status” is overloaded.
