1) Sequence diagram: Archive + Determination Signature + Verification

sequenceDiagram
  autonumber
  participant UI as Client/UI
  participant API as API Gateway
  participant CS as Case Service
  participant ES as Event Store (append-only)
  participant WORM as WORM Object Store (immutable blobs)
  participant KMS as Key Service / KMS (Ed25519)
  participant VR as Verifier (internal/external)

  %% Pre-archive readiness
  UI->>API: GET /api/cases/:id (includes prerequisites summary + version)
  API->>CS: Load case projection
  CS->>ES: Read events (case stream)
  ES-->>CS: Events
  CS-->>API: Case snapshot + caseStateHash + version
  API-->>UI: 200 OK

  Note over UI: UI validates 5/5 prerequisites met<br/>and confirmation prior to archive

  %% Archive request (idempotent)
  UI->>API: POST /api/cases/:id/archive (idempotencyKey)
  API->>CS: ArchiveCase(cmd)
  CS->>ES: Append CASE_ARCHIVE_REQUESTED (idempotencyKey, actor, reason)
  ES-->>CS: OK (eventNumber)

  CS->>ES: Append CASE_ARCHIVED (archivedAt, reasonCode, reasonNote, prereqDigest, caseStateHash, priorVersion)
  ES-->>CS: OK

  CS-->>API: ArchiveResult(state=ARCHIVED, caseStateHash, version++)
  API-->>UI: 204 No Content (or 200 with state)

  %% Determination creation after archive (should be blocked if policy)
  Note over UI,API: If determinations allowed only pre-archive,<br/>API returns 409/403 and UI switches to read-only

  %% Determination issuance (when allowed by policy)
  UI->>API: POST /api/cases/:id/determinations (payload)
  API->>CS: CreateDetermination(cmd)
  CS->>ES: Read case projection (must match expectedVersion)
  ES-->>CS: Events
  CS->>CS: Compute contentHash + caseStateHash
  CS->>KMS: Sign (contentHash || caseStateHash || canonicalFields)
  KMS-->>CS: signatureB64 + publicKeyId

  CS->>WORM: Put determination JSON/PDF (content-addressed by contentHash)
  WORM-->>CS: objectUri (immutable)

  CS->>ES: Append DETERMINATION_ISSUED (contentHash, signatureB64, publicKeyId, caseStateHash, objectUri)
  ES-->>CS: OK
  CS-->>API: DeterminationRef
  API-->>UI: 201 Created

  %% Verification workflow (external or internal)
  VR->>API: GET /api/determinations/:id (or objectUri)
  API->>CS: Load determination ref + objectUri
  CS->>ES: Read determination events
  ES-->>CS: Events
  CS-->>API: Determination metadata + objectUri + signing fields
  API-->>VR: 200 OK

  VR->>WORM: Fetch content (immutable)
  WORM-->>VR: content bytes / canonical JSON
  VR->>VR: Recompute contentHash
  VR->>VR: Verify Ed25519 signature using publicKeyId
  VR->>API: GET /api/cases/:id/state-hash?version=...
  API-->>VR: caseStateHash (recomputed from event stream)
  VR->>VR: Validate (hashes + signature + context)

Implementation notes embedded in the diagram
	•	Idempotency: archive should accept a stable idempotencyKey and return 200/204 if the same key was previously applied to the same case. A 409 should be reserved for true logical conflicts (e.g., expectedVersion mismatch without an idempotency key).
	•	State binding: sign over contentHash + caseStateHash (+ canonicalFields) to ensure the signature is invalid if either the document content or the underlying case state changes.
	•	WORM separation: event store holds the fact; WORM store holds the artifact; events reference immutable objectUri and contentHash.

Key schema decisions (aligned with your constraints)
	•	Hashing: contentHash should be computed over a canonical form of the evaluation payload (e.g., JSON Canonicalization Scheme), excluding signature and any non-deterministic fields.
	•	Context binding: caseStateHash is first-class and signed, preventing “valid evaluation attached to wrong case version.”
	•	WORM references: objectUri is immutable; events refer to it; no deletes.
