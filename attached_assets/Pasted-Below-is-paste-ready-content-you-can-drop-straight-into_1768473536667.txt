1) CANON RULEBOOK v4.0 (Public Spec)

Canon v4.0 — Outcome-Blind Procedural Review

Purpose: Determine whether a decision is procedurally reviewable and whether a procedurally valid determination can be made without using outcome knowledge.

Core commitments:
	•	Outcome-blind: outcomes are not admissible to judge what “should have been known.”
	•	Time-locked: evaluation is performed relative to a defined decision-time boundary.
	•	Grounded: only documents explicitly included in the case record are admissible.
	•	Auditable: every determination produces a receipt listing evidence used, rules applied, and missing elements.

Definitions
	•	Decision Target: A single, answerable governance question (e.g., “Was disciplinary action procedurally permissible based on documentation available at time T?”).
	•	Decision Time (T): The timestamp that defines what information is admissible.
	•	Admissible Evidence: Documents or structured evidence entries that existed at or before T (or explicitly designated as contemporaneous to T).
	•	Procedural Determination: One of:
	•	Permitted (sufficient procedural basis to proceed)
	•	Advisory Only (insufficient basis for high-stakes action, but can issue guidance)
	•	Blocked (cannot proceed due to missing required elements)

Canon Evaluation Sequence (Mandatory)
	1.	Lock Decision Target + Decision Time
	2.	Apply Gatekeeper (Admissibility)
	3.	Assess Evidence Sufficiency (Readiness Checklist)
	4.	Generate Determination
	5.	Issue Determination Receipt (hashes + citations)

Readiness Checklist (v4.0)

The case is evaluated on 5 conditions. Each condition has minimum requirements. The system shows:
	•	Met / Missing
	•	Count needed (equation format)
	•	Which evidence items satisfied it

Condition A — Decision Target Defined (Required Gate)

Requirement: One Decision Target set.
	•	If missing → Blocked.

Condition B — Temporal Verification (2-of-2 required)

Requirement: Both must be satisfied:
	1.	Decision-time timestamp on at least one key evidence artifact (e.g., record header, metadata, signature, system log)
	2.	Sequence of events established (timeline with at least 2 ordered events anchored to timestamps)

If missing → readiness cannot exceed “Advisory Only.”

Condition C — Independent Verification (1-of-1 required)

Requirement: At least one independent corroborator:
	•	1 usable interview (defined below), OR
	•	1 independent document (e.g., third-party report, device log, external audit note)

Usable interview standard:
	•	Identified interviewer + date/time + role
	•	Verbatim or structured Q/A (not pure narrative)
	•	Signed/attested or recorded chain-of-custody
	•	Clear linkage to a claim in the case

Condition D — Policy Application Record (1-of-1 required)

Requirement: At least one record showing how policy was applied at decision time:
	•	policy excerpt + applicable section(s)
	•	mapping from facts to policy criteria
	•	who applied it + when + what action was taken/proposed

Condition E — Contextual Constraints (1-of-1 required)

Requirement: At least one artifact describing constraints at decision time:
	•	staffing/resource constraints OR time pressure OR system conditions
	•	what was reasonably accessible/known at the time

Determination Logic (v4.0)
	•	Blocked if Decision Target missing OR Gatekeeper fails (no admissible evidence).
	•	Decision Permitted if ≥3 of 5 conditions met AND Temporal Verification is met (2/2) AND Decision Target defined.
	•	Advisory Only if <3 of 5 met OR Temporal Verification is not met.
	•	Note: “≥3 of 5” is a default policy setting. It must be configurable per customer, but the receipt must record the threshold.

Outputs Required

Every run must produce:
	1.	Checklist Status (Met/Missing, counts)
	2.	What’s Missing (Equation Form)
	3.	Procedural Determination
	4.	Determination Receipt (hash-anchored, signed, exportable)

    "min_conditions_met": 3,
    "total_conditions": 5,
    "temporal_required": true
  },
  "documents_considered": [
    {
      "doc_id": "doc_01",
      "filename": "Policy_Document.pdf",
      "sha256": "…",
      "uploaded_at": "…",
      "version": "v1.0"
    }
  ],
  "admissibility": {
    "rule": "Only evidence available at/before decision_time is admissible",
    "admitted_doc_ids": ["doc_01", "doc_02"],
    "excluded_doc_ids": [],

  },
  "checklist": {
    "A_decision_target_defined": {
      "required": true,

    "B_temporal_verification": {
      "required": true,
      "met": true,

    "conditions_met": 3,

    "Independent Verification: need (1 usable interview) OR (1 independent document)",
    "Policy Application: need (1 policy application record)"
  ],
  "case_state_hash": {
    "sha256": "hash_of_all_admitted_doc_hashes+decision_target+decision_time+checklist_results"

    "algorithm": "ed25519",

    "signature_b64": "…"
  }

4) GAP EQUATION ENGINE (What user asked for)

Implement a deterministic “needs expression” per condition.

	•	Temporal Verification: 2/2 met or need: 1 decision-time timestamp + 1 ordered timeline event

5) BACKEND / DATA MODEL

Tables
	•	cases
	•	documents (id, case_id, filename, version, sha256, uploaded_at, metadata_json)
	•	case_events (timeline)
	•	decision_targets (case_id, text, set_at, set_by, locked_at)
	•	determinations (case_id, receipt_json, status, created_at, case_state_hash, signature)

Hashing
	•	On upload: compute SHA-256 of file bytes.
	•	case_state_hash = sha256(concat(
canon_version,
decision_target_text,
decision_time_iso,
admitted_doc_sha256s_sorted,
checklist_results_json_canonical
))

Signing
	•	Use server-side ed25519 key pair.
	•	Sign the canonical JSON serialization of receipt without signature field.
	•	Store signature + key id in receipt.
