You are right: that screen is optimized for an already-in-progress procedural audit, not for the moment a user selects a case. As a “first page,” it violates basic information architecture:
	•	It does not answer “What is this case?” within 3 seconds.
	•	It forces the user into question-led navigation before they’ve built a mental model.
	•	It buries the global state (what exists, what’s missing, what’s next) behind UI fragments.

The result is exactly what you described: it feels like you need a PhD to orient yourself.

What the first page should do instead

When a user clicks a case, the landing view should be a Case Overview that answers—immediately:
	1.	What is the decision target? (or “not set”)
	2.	Where are we in the workflow? (Intake → Review → Determine → Closure)
	3.	What prerequisites are satisfied vs missing? (the 5 prerequisites, as a single readable strip)
	4.	What evidence exists? (docs, timeline anchors, independent verification artifacts, policies)
	5.	What is the single next best action? (one big primary CTA)

Everything else (question lists, deep audit panels) should be secondary, behind “Drill in.”

The specific UI problem in your screenshot

That “What do you want to know?” + grouped question cards (“Permission to Decide,” “Evidence Status,” etc.) is a help layer. It’s useful once you know the case. But as a landing page it’s backwards: it asks for a question before the user understands the terrain.

It also competes with the right-side “Decision Readiness” panel, creating two centers of gravity.
