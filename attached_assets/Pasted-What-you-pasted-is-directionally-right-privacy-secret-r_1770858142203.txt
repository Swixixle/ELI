## 1) Rotate secrets (do this even if you think nothing leaked)

### Generate a new ingest token (local or Replit shell)

Use one of these:

```bash
openssl rand -base64 32
```

or (more URL-safe):

```bash
openssl rand -hex 32
```

### Set in Replit (ELI deployment)

In **Deployments → (your Autoscale deployment) → Secrets/Env**:

* `ELI_INGEST_TOKEN = <new token>`
* `ELI_INGEST_SOURCES = lantern` (or your allowlist)

Then **redeploy** so the new env is active.

### Set in Lantern deployment

In Lantern’s deployment secrets:

* `ELI_BASE_URL = https://ajmaksimeli.com`
* `ELI_INGEST_TOKEN = <same new token>`

**Important:** Never place tokens in client-side code. Keep them server-only.

## 2) Verify secrets did not end up in Git history

Making a repo private in GitHub helps, but it **does not “unpublish” old commits** from everywhere they might have been mirrored.

### Quick local scan (from repo root)

```bash
git grep -n "ELI_INGEST_TOKEN\|Bearer \|SECRET\|API_KEY\|PRIVATE_KEY" .
```

If you ever committed a real token/key:

* assume it’s compromised
* rotate it (you’re doing that anyway)
* then **rewrite history** (see below)

### If secrets were committed: rewrite history

Use `git filter-repo` (preferred) to remove the secret from all commits, then force-push. This is only needed if you actually committed secrets.

High-level steps:

1. Remove secrets from files
2. Rewrite history to purge them
3. Force push
4. Rotate again (because the old value existed publicly)

## 3) Confirm the “public DB” risk realistically

A Replit “public deployment” **does not automatically expose the database**. The DB is accessible through your app’s server runtime, not directly crawled, **unless**:

* you had endpoints that dump schema/data
* you had an admin/debug route open
* you logged sensitive data into public logs
* you exposed backups/snapshots in static files

So the real question isn’t “was the DB public,” it’s:

* **Did any route allow unauthorized reads?**
* **Did logs leak anything?**

---

## 4) Tighten ingress on the ingest endpoint

Even with a token, add small guardrails:

### A) Rate limit

* per IP
* per source
* per token
  This stops brute forcing and accidental loops.

### B) Return “duplicate” as a safe success

For your dedupe ledger: return 409 *with* the original `{ caseId, url }`.
This makes Lantern retries safe and prevents thundering herds.

### C) Audit the ingest endpoint itself

You already do audit logging—keep:

* `source`
* `requestId`
* `caseId`
* `result` (created/deduped/rejected)
* and a short rejection reason

---

## 5) Log review: look for obvious abuse signals

In production logs, search for:

* unexpected `401/403` spikes on ingest
* unusual volumes of `409 DUPLICATE_REQUEST` you didn’t generate
* requests without the allowlisted `source`
* any ingest calls outside your own testing windows

If you see weirdness, the response is simple: rotate token again + temporarily disable ingest (feature flag).
